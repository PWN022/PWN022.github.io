# 常规文件上传

```
一定要明白：
无文件解析安全问题上，格式解析是一对一的（不能jpg解析php）
换句话来说有解析错误配置或后缀解析漏洞时才能实现格式差异解析

文件上传安全指的是攻击者通过利用上传实现后门的写入连接后门进行权限控制的安全问题，对于如何确保这类安全问题，一般会从原生态功能中的文件内容，文件后缀，文件类型等方面判断，但是漏洞可能不仅在本身的代码验证逻辑中出现安全问题，也会在语言版本，语言函数，中间件，引用的第三方编辑器等存在缺陷地方配合利用。另外文件上传也有多个存储逻辑，不同的文件存储方案也会给攻击者带来不一样的挑战！

常遇到的问题：
无限制：

限制后缀格式（判断后缀名（黑白名单），MIME类型，文件头等）
```

## .htaccess

一般上传用的不多，主要用来留后门  

必须是在`apache`中，且`.htaccess`文件可以上传并执行 ，这个文件定义了哪些格式文件可以以脚本方式解析

```
AddType application/x-httpd-php.png
```

借助这个配置让`.png`的命名文件当作类型application/x-httpd-php执行（php的类型）

## MIME类型

## 文件头判断

前提：

```
Content-Type：image/png
```

常见文件头标志：

```
JPEG(jpg),文件头：FFD8FF,通常以 `FF D8` 开始
PNG(png),文件头：89504E47,其中 `50 4E 47` 是 "PNG" 的ASCII码
GIF(gif),文件头：47494638,前6字节 `47 49 46 38` 是 "GIF8" 的ASCII码
...等
```

## 逻辑不严-条件竞争

按逻辑来说，一般文件上传是先验证合法性，再进行上传

如果出现了先进行上传后验证的，可以使用这种方法

```
<?php fputs(fopen('xiao.php','w'),'<?php eval($_REQUEST[1]);?>');?>
```

**上传不断发包**

不断上传携带了后门代码的文件

进行上传携带了后门代码的数据包中，修改部分为变量，一直循环，达到不断发包的目的（比如修改user-agent在不影响发包的情况下，任意数字设置为变量，payload设置为无限循环）

**请求不断发包**

不断访问携带了后门代码的文件

在准备好上一步骤时，再对已知上传的路径/文件名进行不断访问（同样修改数据包中的内容），使得触发上一步上传携带的后门代码

## 二次渲染
