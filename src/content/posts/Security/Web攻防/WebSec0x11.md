CSRF，全称：Cross-site request **forgery** 跨站请求**伪造**

解释说明：

用户登录过受信任的网站A->网站A验证通过，返回**Cookie**

这时用户不小心访问了恶意网站B->恶意网站B要求访问网站A->用户访问网站A，并执行了网站B携带的恶意代码

这时攻击者就拿到了该用户的**Cookie**

CSRF和XSS的区别：XSS一般是网站存在漏洞，允许脚本注入等行为，利用的是**代码层面的漏洞**（过滤不严）。而CSRF是不需要找注入点之类，利用的是**业务逻辑层面的漏洞**（操作未做二次确认）。

# 案例-CSRF利用-同源策略防护

**检测请求头中的Referer字段是不是同一域名或者ip下**

## 规则匹配绕过问题（代码逻辑不严谨）

### 判断同源

还有两种判断方式：

1. 匹配判断
2. 等值判断

一般的只会采用匹配判断因为需要保证功能正常（因为存在路由/域名/静态资源/或者端口问题），除非是个人网站等等才会使用等值

匹配机制改为同源即可绕过：

```
Origin：http://xx.xx.xx.xx/http://xx.xx.xx.xx
Referer：http://xx.xx.xx.xx/http://xx.xx.xx.xx
```

或者

```
Referer: http://xx.xx.xx.xx.attacker.com
```

### 置空来源

以上同源策略的数据包，需要在用户端修改，触发条件比较苛刻，从实战场景来讲是没法操作受害者浏览器改数据包的，所以可以用如下方法绕过：

抓包，修改生成的CSRF POC，在其中添加：

```html
置空来源：<meta name="referrer" content="no-referrer">
```

或者在攻击者的服务器中创建目录为http://xx.xx.xx.xx，然后把CSRF POC放进来

## 配合文件上传绕过（严谨使用同源绕过）

同上，文件上传POC，符合“同源”要求（因为就是自己的域名）

## 配合存储XSS绕过（严谨使用同源绕过）

同上，XSS直接在评论或帖子中注入恶意脚本，脚本内容会自动向同站点发起请求

# 案例-CSRF利用-Token校验防护

token(令牌,也可以理解为暗号,在数据传输 之前，要先进行暗号的核对，暗号不一致则拒绝数据传输)

CSRF_token 对关键操作增加Token参数，token必须随机，每次都不一样，一般**存储在Session中**（服务端），或者作为表单隐藏字段直接下发到页面，但是也可以存储在Cookie中，与验证码一样。

解释说明：

用户登录网站A，服务器生成一个随机Token，存入Session，并在表单页面中埋入该Token->用户在没有退出网站A的情况下访问了恶意网站B

网站B构造恶意请求，试图向网站A提交表单->网站A收到请求，检查请求中的Token，从Session中取出正确的Token，与请求中携带的Token进行比对，因为恶意请求中**无法获取Token**，所以Token不匹配 ->服务器拒绝请求 

**易混淆点**：

第一种：用户**本地确实有Token，但它不在Cookie里（或即使放在Cookie里，攻击者也无法通过CSRF获取）**，攻击者构造的请求拿不到，所以服务器会拒绝。

第二种：**而且即使通过模拟对方的数据包**（本地搭建同款网站），本地生成的Token与用户A的Token不同，同样会导致失败。

第三种：**如果拿到了用户A的token**，**在Token失效之前**，攻击者可以成功伪造请求。但若服务器采用了**一次性Token**机制（每次请求后立即刷新），则该Token在下次请求时失效，攻击者无法持续利用。
